Draw a system architecture diagram for a local web application called resman, a research vault manager that runs on a single Linux desktop machine.

The system has three main layers. The bottom layer is the filesystem. On disk there is a top-level research folder that contains multiple vault subfolders. Each vault is an Obsidian wiki project folder with its own notes, raw sources, wiki pages, and a CLAUDE.md config file. There is also a system config folder containing YAML files for settings and schedules, a task log file, and a tools folder with shell scripts. A vault template folder is used to clone new vaults.

The middle layer is a Python web server running locally. It is built with Flask and Socket.IO and uses eventlet for async handling. Inside the server there are several modules. The vault registry reads the config YAML and knows about all vaults. The vault runtime launches terminal sessions for each vault. The task manager maintains a priority queue of operations to run on vaults, stores their history as an append-only event log, and supports parent tasks that fan out into child tasks when an operation runs across all vaults. The window state module tracks whether a Claude Code session is currently active, between sessions, or the weekly period has ended, based on manual time sync input from the user. The scheduler fires recurring cron tasks using APScheduler but only when the window state is active. The tmux manager creates and destroys isolated tmux sessions on a custom socket. The PTY bridge forks a pseudo-terminal attached to a tmux session and streams its output to the browser over a WebSocket connection.

The top layer is the browser UI running on localhost. It is a single-page application with no build step, using vanilla JavaScript, xterm.js for terminal rendering, and Socket.IO for real-time communication. The UI has three visual zones. On the left is a narrow sidebar listing all research vaults as rows with colored status dots showing whether each vault has an active session, a running task, or is idle. Below the registered vaults a divider separates any discovered but unregistered vault folders, each with a register button. At the bottom of the sidebar are buttons to create a new vault and to open the config editor. In the center is a large main panel with tabs across the top. The Terminal tab shows one or more xterm.js terminal windows connected live to tmux sessions running inside the selected vault, with buttons to start a bash shell, start a Claude Code session, or open Obsidian for that vault. The Docs tab shows a markdown viewer and editor for vault documentation. The Tasks tab shows the task queue with rows for each pending, running, deferred, or completed task, with expandable parent rows that reveal per-vault child task status when a task was run across all vaults. The Config tab has a YAML text editor for the system config files. Along the very bottom of the screen is a persistent status bar showing the current window state, a countdown timer when a window is active, and a sync dropdown menu where the user manually tells the system when a Claude window started, how long it has, or that it ended.

The arrows in the diagram should show the browser connecting over HTTP and WebSocket to the Flask server on localhost port 5090. The Flask server talks to tmux via shell commands. The PTY bridge streams tmux output back to the browser. The task manager reads and writes JSONL files on disk. The vault runtime launches claude and bash processes inside the vault folders on disk. The scheduler module connects to the task manager to fire cron tasks. The window state module gates whether the task manager is allowed to dispatch new tasks.

The visual style should be a clean technical architecture diagram with labeled boxes and arrows, dark background, color-coded layers for filesystem, server, and browser, and icons or small illustrations for the terminal windows, vault folders, and the Obsidian app.
